# dcnet_progress — Business Process Management for ERPNext

**Date:** 2026-04-12
**Status:** Approved
**Phase:** 1 (Core Engine + Visual Designer)

## 1. Overview

dcnet_progress is a no-code BPM (Business Process Management) app for ERPNext, inspired by Misa Amis "Quy trinh" (the most modern version). It provides a visual process designer, form builder per step, parallel/conditional execution, and a full process runtime — independent from Frappe's built-in Workflow.

### Why not extend Frappe Workflow?

| Aspect | Frappe Workflow | dcnet_progress |
|--------|----------------|----------------|
| Architecture | State machine on document | Independent graph engine |
| Binding | 1:1 with a DocType | Process has its own forms, not tied to any DocType |
| Parallel steps | No (single `workflow_state` field) | Fork/Join via graph model |
| Custom forms | No — uses DocType form | Form builder per step |
| Process runs | No concept — document IS the instance | Many runs from 1 definition |
| Visual designer | No (table config) | React Flow drag-and-drop |
| Analytics | No | Dashboard: backlog, overdue, avg time |

**Phase 2 integration:** Process completion will auto-create ERPNext documents (PO, Payment Entry, etc.) which then follow their own Frappe Workflow.

## 2. Data Model

### 2.1 Process Category

Standalone DocType for organizing process definitions.

```
Process Category
├── category_name (Data, required, unique)
├── description (Small Text)
└── icon (Data, optional — Ant Design icon name)
```

### 2.2 Process Definition

Core DocType storing the process blueprint. NOT submittable (versioning handled manually via `version` field + `status`).

```
Process Definition
├── title (Data, required)
├── description (Text Editor)
├── category (Link → Process Category)
├── version (Int, default 1, read-only — auto-increment on publish)
├── status (Select: Draft | Published | Suspended)
├── graph_data (JSON, hidden) ← full React Flow state (nodes, edges, viewport)
├── owner (Link → User — designer who created it)
│
├── [Child Table] Process Step
│   ├── step_id (Data, required) ← unique within definition, e.g. "step_001"
│   ├── step_type (Select: Start | Task | Approval | End)
│   ├── label (Data, required)
│   ├── description (Small Text)
│   ├── form_schema (JSON) ← array of field definitions (see §2.5)
│   ├── executor_type (Select: Initiator | User | Role | Direct Manager | Previous Step Executor)
│   ├── executor_value (Data) ← user email, role name, or step_id depending on type
│   ├── allow_reassign (Check, default 0)
│   ├── allow_return (Check, default 0)
│   ├── allow_forward (Check, default 0)
│   ├── position_x (Float) ← React Flow node X
│   └── position_y (Float) ← React Flow node Y
│
└── [Child Table] Process Transition
    ├── transition_id (Data, required) ← unique, e.g. "edge_001"
    ├── from_step (Data, required) ← step_id reference
    ├── to_step (Data, required) ← step_id reference
    ├── trigger (Select: On Complete | On Reject) ← default "On Complete"
    ├── condition_type (Select: Always | Field Condition)
    ├── condition_json (JSON) ← {"field": "amount", "op": ">", "value": 20000000}
    └── label (Data) ← display label on edge, e.g. "Số tiền > 20 triệu"
```

**Notes:**
- `step_id` and `transition_id` are string IDs generated by the frontend (UUID or incremental). They are NOT Frappe Link fields — child table rows don't have stable names for linking.
- `Start` and `End` are special step types: Start has no executor (auto-activates), End marks completion.
- `form_schema` is JSON because Frappe doesn't support grandchild tables. See §2.5 for schema format.
- `graph_data` stores the full React Flow serialized state for designer restore (node positions, edge styles, viewport zoom/pan). Redundant with position_x/y on steps but needed for edge routing points and viewport state.

### 2.3 Process Run

Runtime instance of a process definition.

```
Process Run
├── process_definition (Link → Process Definition, required)
├── definition_version (Int) ← snapshot which version was used
├── definition_snapshot (JSON) ← full copy of steps + transitions at run creation
├── status (Select: Draft | In Progress | Completed | Cancelled | Withdrawn)
├── initiated_by (Link → User)
├── completed_at (Datetime)
│
├── [Child Table] Process Run Step
│   ├── step_id (Data) ← matches definition step_id
│   ├── status (Select: Pending | Active | Completed | Skipped | Rejected)
│   ├── assigned_to (Link → User) ← resolved at activation time
│   ├── form_data (JSON) ← data entered at this step
│   ├── action (Select: Approve | Reject | Forward | Return | null)
│   ├── action_comment (Text)
│   ├── forwarded_to (Link → User, optional)
│   ├── activated_at (Datetime)
│   ├── completed_at (Datetime)
│   └── processed (Check, default 0) ← internal flag for engine advance_run()
│
└── [Child Table] Process Run Activity
    ├── step_id (Data)
    ├── user (Link → User)
    ├── action (Data) ← "activate", "approve", "reject", "forward", "return", "withdraw", "reassign"
    ├── comment (Text)
    ├── from_status (Data)
    └── to_status (Data)
```

**Notes:**
- `definition_snapshot` is a full JSON copy of steps + transitions at the time the run was created. This ensures the run is immutable even if the definition is later edited/republished. Without this, editing a published definition could corrupt in-flight runs.
- `Process Run Activity` is the audit trail — every action logged with timestamp (via Frappe's auto `creation` field).

### 2.4 Process Run Step — Status Machine

```
Pending ──[activate]──→ Active ──[approve/complete]──→ Completed
                            │                              
                            ├──[reject]──→ Rejected        
                            │                              
                            ├──[forward]──→ Active (new assigned_to)
                            │                              
                            └──[return]──→ Active (previous step re-activated)
                                                           
Pending ──[skip]──→ Skipped (when parallel branch not taken)
```

    "description": "Nhập số tiền cần thanh toán"
  },
  {
    "fieldname": "reason",
    "label": "Lý do",
    "fieldtype": "Text",
    "required": true
  },
  {
    "fieldname": "department",
    "label": "Phòng ban",
    "fieldtype": "Link",
    "options": "Department",
    "required": true
  },
  {
    "fieldname": "priority",
    "label": "Mức độ ưu tiên",

### 2.6 Condition JSON Format

Transition conditions for field-based routing:

```json
{
  "logic": "AND",
  "conditions": [
    {"field": "amount", "op": ">", "value": 20000000},
    {"field": "department", "op": "==", "value": "Ke toan"}
  ]
}
```

Operators: `==`, `!=`, `>`, `>=`, `<`, `<=`, `in`, `not_in`.

The engine evaluates conditions against the accumulated `form_data` from all completed steps in the run.

## 3. Process Execution Engine

### 3.1 Core Algorithm

```python
def advance_run(run_name):
    """Called after a step is completed. Evaluates transitions and activates next steps."""
    run = frappe.get_doc("Process Run", run_name)
    snapshot = json.loads(run.definition_snapshot)
    
    # Find all just-completed steps
    completed_steps = [s for s in run.run_steps if s.status == "Completed" and not s.processed]
    
    for completed_step in completed_steps:
        # Get outgoing transitions from this step
        outgoing = [t for t in snapshot["transitions"] if t["from_step"] == completed_step.step_id]
        
        for transition in outgoing:
            if evaluate_condition(transition, run):
                target_step = transition["to_step"]
                target_def = get_step_def(snapshot, target_step)
                
                if target_def["step_type"] == "End":
                    # Check if ALL incoming paths to End are complete
                    if all_predecessors_complete(snapshot, target_step, run):
                        run.status = "Completed"
                        run.completed_at = now()
                else:
                    # Check join condition: all incoming edges must be satisfied
                    if all_predecessors_complete(snapshot, target_step, run):
                        activate_step(run, target_step, completed_step)
        
        completed_step.processed = True
    
    run.save()
```

### 3.2 Fork/Join Logic

**Fork:** When a step has multiple outgoing transitions that ALL evaluate to true, multiple next steps are activated simultaneously (parallel execution).

**Join:** When a step has multiple incoming transitions, it is only activated when ALL predecessor steps (connected by incoming edges) are completed. This prevents premature activation.

### 3.3 Executor Resolution

```python
def resolve_executor(step_def, run, previous_step_user=None):
    executor_type = step_def["executor_type"]
    executor_value = step_def["executor_value"]
    
    if executor_type == "Initiator":
        return run.initiated_by
    elif executor_type == "User":
        return executor_value  # specific user email
    elif executor_type == "Role":
        users = frappe.get_all("Has Role", filters={"role": executor_value, "parenttype": "User"}, pluck="parent")
        return users[0] if users else None  # Phase 1: first available. Phase 2: round-robin/least-loaded
    elif executor_type == "Direct Manager":
        employee = frappe.db.get_value("Employee", {"user_id": run.initiated_by}, "reports_to")
        if employee:
            return frappe.db.get_value("Employee", employee, "user_id")
    elif executor_type == "Previous Step Executor":
        return previous_step_user
    return None
```

### 4.4 Key Component: StepForm

Dynamic form renderer that takes `form_schema` JSON and renders Ant Design form fields:

```tsx
function StepForm({ schema, initialData, readOnly, onSubmit }) {
  const [form] = Form.useForm();
  return (
    <Form form={form} initialValues={initialData} disabled={readOnly}>
      {schema.map(field => (
        <FormField key={field.fieldname} definition={field} />
      ))}
    </Form>
  );
}
```

`FormField` maps fieldtype to Ant Design component: Data→Input, Currency→InputNumber, Date→DatePicker, Select→Select, Link→AutoComplete (fetches from Frappe API), etc.

## 5. User Flows

### 5.1 Thiết kế quy trình

1. Người dùng vào `/process/definitions` → click "Tạo mới"
2. Nhập tên, mô tả, danh mục → vào Trình thiết kế
3. Canvas hiển thị node Bắt đầu. Kéo thả thêm node Công việc/Phê duyệt.
4. Kéo đường nối giữa các node → tạo chuyển tiếp.
5. Click node → panel bên phải: cấu hình loại bước, trường biểu mẫu, người thực hiện.
6. Click đường nối → cấu hình điều kiện (Luôn luôn hoặc theo giá trị trường).
7. Nút "Xem sơ đồ" chuyển giữa chế độ chỉnh sửa và xem sơ đồ.
8. "Lưu nháp" tự động lưu. "Xuất bản" xuất bản (tăng phiên bản).

### 5.2 Chạy quy trình

1. Từ `/process/definitions`, click "Chạy" trên quy trình đã xuất bản.
2. Biểu mẫu bước 1 hiện ra (bước đầu tiên sau node Bắt đầu).
3. Điền dữ liệu ban đầu → Gửi.
4. Engine kích hoạt bước tiếp → người được giao nhận thông báo.
5. Người được giao vào `/process` (Việc cần xử lý) → click vào.
6. Xem dữ liệu từ bước trước (chỉ đọc) + biểu mẫu bước hiện tại.
7. Thực hiện hành động: Đồng ý / Từ chối / Chuyển tiếp / Trả về.
8. Engine tiếp tục đến node Kết thúc → Trạng thái = Hoàn thành.

### 5.3 Hành động

| Hành động | Hiển thị trên UI | Ai thực hiện | Hiệu lực |
|-----------|-----------------|--------------|-----------|
| Đồng ý / Hoàn thành | "Đồng ý" / "Hoàn thành" | Người được giao | Bước hoàn thành, engine chuyển tiếp |
| Từ chối | "Từ chối" | Người được giao | Bước = Từ chối. Nếu có transition `on_reject` → đi theo. Ngược lại → Lượt chạy = Đã hủy. |
| Chuyển tiếp | "Chuyển tiếp" | Người được giao (nếu cho phép) | Giao lại cho người khác |
| Trả về | "Trả về" | Người được giao (nếu cho phép) | Quay lại bước trước |
| Thu hồi | "Thu hồi" | Chỉ người khởi tạo | Hủy lượt chạy đang thực hiện |
| Phân công lại | "Phân công lại" | Admin hoặc người được giao | Đổi người thực hiện bước đang hoạt động |

### 5.4 Tổng quan (Dashboard)

- Tổng lượt chạy theo trạng thái (biểu đồ tròn)
- Tồn đọng theo phòng ban (biểu đồ cột)

| Thông báo | "Bạn có việc cần xử lý trong quy trình [tên]" |
| Validation | "Vui lòng nhập [tên trường]", "Quy trình phải có ít nhất 1 bước Kết thúc" |

**Code convention:** All string literals in React components use Vietnamese directly. No i18n wrapper in phase 1 — raw Vietnamese strings. i18n extraction will happen in a later phase when English is needed.

```tsx
// Phase 1: Vietnamese directly
<Button type="primary">Đồng ý</Button>
<Tag color="green">Hoàn thành</Tag>

// Phase 2 (later): i18n wrapper
<Button type="primary">{t('approve')}</Button>
```

## 8. Permissions

| Role | Can do |
|------|--------|
| System Manager | Everything |
| Process Designer | Create/edit/publish definitions, view all runs |
| Process User | Start runs, execute assigned steps, view own runs |
| Process Viewer | View runs and dashboard only |

These are Frappe Roles. Permission rules set on each DocType via standard Frappe permission model.

## 8. Error Handling

- **Definition validation before publish:** Must have exactly 1 Start node, at least 1 End node, all nodes reachable from Start, no orphan nodes, all steps have at least 1 outgoing transition (except End).
- **Executor resolution failure:** If no user found (e.g., no Direct Manager in HR), step stays Active with `assigned_to = null` → notification to Process Designer + System Manager.
- **Concurrent execution:** Process Run uses Frappe's document locking (`frappe.lock_doc()`) during step execution to prevent race conditions on parallel approval.
